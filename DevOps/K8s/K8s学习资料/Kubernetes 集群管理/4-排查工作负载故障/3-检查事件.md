# 检查事件

我们知道它「失败了」，但不知道「为什么」。为了找出原因，我们需要近距离观察。

`kubectl describe` 命令就是你的放大镜。它会显示特定资源的内部事件和日志。

**Events**（事件）部分非常有价值，因为它保留了来自调度器（scheduler）和 kubelet 等 Kubernetes 组件的简短叙述。当新手不阅读事件就直接尝试随机修复时，往往会浪费时间去解决错误的问题。

检查损坏的 Pod：

```bash
kubectl describe pod broken-pod
```

这会输出大量数据。滚动到底部找到 **Events** 部分。

你应该会看到类似以下内容：

```plaintext
Type     Reason     Message
----     ------     -------
Normal   Pulling    Pulling image "nginx:wrongtag123"
Warning  Failed     Failed to pull image "nginx:wrongtag123": manifest unknown
Warning  Failed     Error: ErrImagePull
```

消息 `manifest unknown` 是关键证据。这意味着 Docker 仓库（镜像存储的地方）中没有 `nginx:wrongtag123` 的记录。

案件告破：镜像标签无效。
