# 驱逐节点

封锁只会阻止「新」的 Pod。房间里仍然有正在运行的应用程序（房客）。为了安全地移除它们，我们使用 `drain` 命令。

该命令会驱逐 Pod。随后，Kubernetes 会意识到副本数量（2 个）未达标，并尝试在其他可用节点上创建新的替代 Pod。

这里的「驱逐」（evict）一词很重要。驱逐并不是一种粗暴的停止。它以 Kubernetes 可感知的方式请求工作负载离开，以便控制器可以替换它们，并遵守终止规则。这就是为什么 `drain` 比手动删除节点上的随机进程更安全的原因。

执行驱逐命令。我们需要添加一些标志来强制在此环境中进行驱逐：

- `--ignore-daemonsets`：某些系统 Pod（DaemonSet）与节点绑定，无法移动。我们忽略它们。
- `--delete-emptydir-data`：如果 Pod 使用了本地临时存储，驱逐将删除该数据。我们确认此操作。
- `--force`：删除非 ReplicationController、ReplicaSet、Job、DaemonSet 或 StatefulSet 管理的 Pod（例如 Minikube 中的 `storage-provisioner`）时需要此参数。

```bash
kubectl drain $(kubectl get nodes -o jsonpath='{.items[0].metadata.name}') --ignore-daemonsets --delete-emptydir-data --force
```

你将看到指示 Pod 正在被驱逐的输出。

**注意**：在这个单节点的 Minikube 实验中，当我们驱逐「唯一」的节点时，Pod 会被驱逐但会变为 `Pending`（挂起）状态。这是因为没有「其他房间」可以转移。在真实的多节点集群中，它们会立即移动到健康的节点上。
