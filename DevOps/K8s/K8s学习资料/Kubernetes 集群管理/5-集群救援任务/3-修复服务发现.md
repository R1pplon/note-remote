# 修复服务发现

成功让 Pod 运行起来了，干得漂亮！然而，前端团队报告说他们仍然无法访问后端服务。看起来服务发现机制坏了。

Kubernetes Service 依赖于「选择器」（selector）来确定将流量发送到哪些 Pod。如果选择器与 Pod 上的标签不匹配，Service 将指向虚无。

## 任务

1. 检查 `rescue-mission` 命名空间中的 `backend-svc` Service。
2. 将 Service 的 `selector` 与 `backend` Pod 上的 `labels` 进行比较。
3. 编辑 `service.yaml` 文件以正确选择 `backend` Pod。
4. 应用更新后的配置。
5. 验证 Service 现在是否拥有 Endpoints。

## 要求

- Service 名称 `backend-svc` 必须保持不变。
- Service 必须正确指向带有标签 `app: backend` 的 Pod。

## 示例

验证服务是否有 Endpoints：

```bash
kubectl get endpoints backend-svc -n rescue-mission
```

预期输出：

```plaintext
NAME          ENDPOINTS         AGE
backend-svc   10.244.0.5:80     5m
```

## 提示

<details>
<summary>如何检查 Pod 标签</summary>

使用 `kubectl get pods -n rescue-mission --show-labels` 检查 Pod 标签。

</details>

<details>
<summary>如何修复服务选择器</summary>

更新 `service.yaml`，使 `selector` 与 Pod 的标签（`app: backend`）匹配。然后重新应用该文件。

</details>
