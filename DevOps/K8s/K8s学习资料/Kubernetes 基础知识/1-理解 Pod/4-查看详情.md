# 查看详情

在这一步中，你将学习如何“深入底层”。`get` 命令为我们提供了摘要，但有时你需要确切了解发生了什么——尤其是在出现问题时。

`describe` 命令提供了关于 Kubernetes 资源的详细报告。

对于初学者来说，将 `kubectl get` 和 `kubectl describe` 视为两个不同层级的观察方式会很有帮助。`get` 回答的是“当前的总体状态是什么？”，而 `describe` 回答的是“我们是如何走到这一步的？”。事件历史记录通常是找到根本原因的最快途径。

运行以下命令来检查 `nginx-pod`：

```bash
kubectl describe pod nginx-pod
```

这将输出大量信息。滚动到底部并查看 **Events** 部分。它按时间顺序讲述了 Pod 的启动过程：

```plaintext
Events:
  Type    Reason     Age   From               Message
  ----    ------     ----  ----               -------
  Normal  Scheduled  ...   default-scheduler  Successfully assigned default/nginx-pod...
  Normal  Pulling    ...   kubelet            Pulling image "nginx:latest"
  Normal  Pulled     ...   kubelet            Successfully pulled image "nginx:latest"...
  Normal  Created    ...   kubelet            Created container nginx
  Normal  Started    ...   kubelet            Started container nginx
```

此事件日志是你进行调试的第一站。如果 Pod 卡在 `Pending` 或 `Error` 状态，Events 部分通常会告诉你原因（例如，“Insuficient CPU”或“ImagePullBackOff”）。

现在你已经创建、验证并检查了你的第一个 Kubernetes Pod！
