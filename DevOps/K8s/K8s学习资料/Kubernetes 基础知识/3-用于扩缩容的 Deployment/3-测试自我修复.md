# 测试自我修复

Kubernetes 最强大的功能之一是 **自我修复（Self-Healing）**。Deployment 保证实际的 Pod 数量始终与期望的数量（3）匹配。如果有一个丢失，它会立即替换它。

这是因为 Kubernetes 会不断地协调实际状态与期望状态。删除一个 Pod 并不会改变存储在 Deployment 中的期望副本数，因此系统会将删除解释为偏差，并创建一个替代品。

让我们通过手动删除其中一个 Pod 来测试这一点。

首先，再次列出 Pod：

```bash
kubectl get pods
```

复制其中 **一个** Pod 的完整名称（例如 `nginx-deployment-66b6c48dd5-8j64w`）。

现在，删除该特定的 Pod：

```bash
kubectl delete pod <paste-pod-name-here>
```

Kubernetes 会提示该 Pod 已被删除。立即再次运行 `kubectl get pods`：

```bash
kubectl get pods
```

你会看到神奇的一幕：

1. 旧的 Pod 正在终止（或已消失）。
2. 一个名称不同且 Age 非常小（例如 2 秒）的 **新** Pod 已经出现！

Deployment 注意到少了一个工作者，并立即雇佣了一个新的，以保持团队规模为 3。这确保了即使个别服务器或容器发生故障，你的应用程序也能保持在线。
